# OrdoStudio NYC Pilot

Status: first wedge product workflow as of 2026-05-13

This workflow uses OrdoStudio to sell, support, and improve OrdoStudio.

The pilot is intentionally concrete:

```text
NYC tech meetup
-> QR on homepage
-> tracked visitor session
-> 30-day hosted Ordo offer
-> 10 active trial spots
-> user accepts offer
-> hosted trial Ordo is provisioned
-> user can back up and restore
-> trial expires or resets
-> user can request a strategy session
-> handoff reaches Keith in Support
-> Studio produces 10-30 second promo videos
-> videos are staged or published
-> Growth measures response
-> feedback and referrals can earn more hosted time
```

This is not just a marketing campaign. It is the first compact proof of the
full product shape: Growth, Offers, Access, Systems, Support, Studio, Knowledge,
and Member View all participate in one bounded loop.

## Current Demo Delivery Lane

The current 0.1.9 demo chain is narrower than the full pilot loop. It should
prove the product shell and appliance trust boundary before broader rewards,
Executor import, pack assurance, or SUMO reporting work.

Execute in this order unless the owner changes demo priority:

1. Story Intake evidence links to workflow compilation evidence.
2. Studio Preview shows governed workflow state.
3. A tracked public entry creates or returns a first-user relationship handoff.
4. Approved generated-content memory candidates produce a read-only promotion
   readiness packet.

This chain should prove:

- public story path;
- Studio workflow state;
- support handoff path;
- memory readiness without automatic promotion;
- evidence-safe UI;
- local appliance trust boundary.

The tracked-entry handoff path is demo-critical for NYC. It should come before
memory readiness when choosing between #419 and #417.

Do not broaden this lane into Executor import, graph import/export, live
provider calls, publishing automation, vector databases, generalized HITL,
canonical Request storage rewrites, full pack assurance, operational reviewer
personas, or consensus review.

## Offer Shape

The default offer:

- 30-day hosted Ordo trial;
- 10 active hosted trial spots;
- reset or wipe after the trial window unless extended or converted;
- backup/export before wipe;
- restore path for the user's Ordo data;
- experimental hosted-service expectations;
- option to request a strategic consulting session with Keith;
- option to submit feedback and earn hosted-time credits;
- option to refer others and earn hosted-time credits.

The product should make scarcity concrete without becoming gimmicky. "10 pilot
spots" should map to actual capacity state, waitlist state, and trial lifecycle
records.

## Rewards Policy

Pilot default:

```text
qualified referral: +7 hosted days
accepted feedback: policy-defined hosted days
maximum extension: policy-defined cap
```

A referral should qualify when the referred person activates a trial or reaches
another owner-approved qualification event. A QR scan alone should not earn the
reward.

Feedback should qualify when it is useful enough to tag, review, or turn into a
product issue, test, doc fix, design correction, or support insight.

Rewards are ledgered. The hosted-time benefit is granted through Access.

## Support Handoff

The strategic consulting session is the first live handoff demo.

Flow:

1. Trial user asks Ordo for Keith or for strategy help.
2. Ordo checks availability, policy, and request context.
3. Ordo asks screening questions if needed.
4. Ordo creates a Support handoff item with conversation evidence.
5. Keith sees the request in Support.
6. Keith accepts, declines, schedules, or asks Ordo to continue screening.
7. The decision and receipt are recorded.

The demo should prove that Ordo can talk anytime while human attention remains
policy-gated.

## Studio Promo Workflow

The pilot should produce short promotional artifacts:

- 10-30 second vertical videos;
- captions and titles;
- platform descriptions;
- thumbnail or frame choices;
- publish/stage metadata;
- performance report.

Direct TikTok and YouTube publishing can come later if OAuth, approval, and
platform policy are ready. The first version can stage downloadable artifacts
with metadata and record whether the owner manually published them.

## Experimental Hosting Expectations

Do not call the pilot a strong SLA. It is an experimental hosted pilot.

The user-facing terms should say, in plain language:

- this is experimental and best-effort;
- do not use the trial as production-critical infrastructure;
- availability is not guaranteed yet;
- AI outputs require human review;
- integrations and publishing paths may fail or change;
- hosted trials may expire, reset, or be wiped after the trial window;
- users should export backups before expiration;
- rewards can be capped, reviewed, rejected, reversed, or changed for abuse;
- leaderboard participation is optional if leaderboards exist;
- external rewards or prizes require explicit terms and approval.

## Success Criteria

The pilot is working when Ordo can show:

- QR scan to visitor session evidence;
- visitor chat to offer acceptance;
- offer acceptance to trial/access state;
- trial slot capacity and waitlist behavior;
- backup/export before expiration;
- hosted-time reward ledger entries for qualified referrals or feedback;
- handoff request visible in Support;
- promo artifact produced in Studio;
- Growth report with scans, chats, acceptances, trial activations, handoffs,
  feedback, referrals, videos produced, and known publication/performance
  evidence.

## Non-Goals

- No payment processing required for the first pilot.
- No cash referral payout automation.
- No public leaderboard by default.
- No guaranteed uptime SLA.
- No direct platform publishing until OAuth and approval gates are ready.
- No reward for scans alone.
